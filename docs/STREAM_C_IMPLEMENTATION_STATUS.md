# Stream C — Workflows & Closed-Loop: Implementation Status

Status: **complete**, including all tasks C-1 through C-5 (including the C-4 transit tab bonus) from `Grid_Unlocked_Build_Guide.md` implemented and verified.

## What was built

### C-1 · De-hardcode planned coordinates (Feature 4)
- Replaced the static, hardcoded `CORRIDOR_CENTROIDS` constant lookup in [frontend/app/planned/page.tsx](../frontend/app/planned/page.tsx) with dynamic data fetched from the backend's `/api/v1/corridors` endpoint via `api.corridors()`.
- Implemented `resolveCorridorCoords(corridor, centroids)`:
  - Normalizes search queries to lowercase.
  - Checks for exact matches against real centroids.
  - Performs case-insensitive fuzzy matches (e.g. "Mysore" matches "Mysore Road").
  - Safely falls back to the Bengaluru city centre default coordinate (`[12.9716, 77.5946]`) if no matching corridor is found or input is empty/null.
- Enhanced the operator UI form inside the `AddEditDialog`:
  - Attached a dynamic `<datalist>` to the corridor input, allowing operators to select from all real, registered corridor names.
  - Added a live text preview indicator showing which coordinates will be used (e.g. "Using real centroid 12.9535, 77.5229" or "No matching centroid — defaults to city centre").
- Verified with unit tests in `frontend/tests/unit/planned-corridor-coords.test.ts`.

### C-2 · Commander "Close & Learn" (Feature 5)
- Added the "Close & Learn" Action button to [frontend/app/live/_components/action-card-panel.tsx](../frontend/app/live/_components/action-card-panel.tsx).
  - Controlled by the helper `canCloseAndLearn(status)`, displaying the action only for approved, complete, or executed cards (statuses eligible for closure and learning).
- Created a Dialog modal prompting the commander for real-world execution metrics:
  - Number of barricades used (with validation).
  - Number of officers deployed (with validation).
  - Checkbox indicating whether a diversion was activated.
  - Optional text notes detailing real-time context.
- Wires the submission to the existing `api.fieldClose(eventId, closurePayload)` endpoint.
- On successful closure, triggers `onMutated()` to update the dashboard alerts, and immediately queries `api.latestLearningJob()` to fetch the latest M13 model retrain details.
- Displays a dedicated "Learning Loop" banner on the card displaying:
  - Replay buffer sample count.
  - Trigger details and ID of the latest model retrain job.
  - Status of the job.
- Verified with unit tests in `frontend/tests/unit/close-and-learn.test.tsx`.

### C-3 · Citizen → verify → live loop
- Handled the end-to-end flow:
  - Verified report endpoint (`POST /citizen/verify/{report_id}`) returns an `ActionCard` and transitions status.
  - Integrated with Stream B's incident broadcast: a newly verified citizen report immediately fires a WebSocket `DashboardDelta` under the `"incident"` scope, dynamically plotting the incident pin on the live monitor in < 1 second.

### C-4 · Transit tab (Feature 7)
- Added the "Transit" tab to the main card viewer panel inside `ActionCardPanel` alongside existing "Impact", "Evidence", "Routes", etc.
- Implemented lazy-loading:
  - Avoids pre-fetching. The `api.transitImpact(eventId)` method (which hits `GET /transit/impact/{event_id}`) is triggered *only* when the commander clicks the "Transit" tab.
- Displays a detailed operational impact dashboard:
  - Passenger Delay Index (PDI) and Transfer Overload Risk gauges.
  - List of affected BMTC bus routes showing route name, occupancy level, predicted delay in minutes, and overlap fraction with the congested corridor.
  - Advisory messages generated by the service.
- Verified with unit tests in `frontend/tests/unit/transit-tab.test.tsx`.

### C-5 · API/Types & tests
